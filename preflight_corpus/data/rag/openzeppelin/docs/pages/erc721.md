# ERC-721

</div>

<div id="content">

<div id="preamble">

<div class="sectionbody">

<div class="paragraph">

We’ve discussed how you can make a *fungible* token using [ERC-20](erc20.html), but what if not all tokens are alike? This comes up in situations like **real estate**, **voting rights**, or **collectibles**, where some items are valued more than others, due to their usefulness, rarity, etc. ERC-721 is a standard for representing ownership of [*non-fungible* tokens](tokens.html#different-kinds-of-tokens), that is, where each token is unique.

</div>

<div class="paragraph">

ERC-721 is a more complex standard than ERC-20, with multiple optional extensions, and is split across a number of contracts. The OpenZeppelin Contracts provide flexibility regarding how these are combined, along with custom useful extensions. Check out the [API Reference](api:token/ERC721.html) to learn more about these.

</div>

</div>

</div>

<div class="sect1">

## Constructing an ERC-721 Token Contract

<div class="sectionbody">

<div class="paragraph">

We’ll use ERC-721 to track items in our game, which will each have their own unique attributes. Whenever one is to be awarded to a player, it will be minted and sent to them. Players are free to keep their token or trade it with other people as they see fit, as they would any other asset on the blockchain! Please note any account can call `awardItem` to mint items. To restrict what accounts can mint items we can add [Access Control](access-control.html).

</div>

<div class="paragraph">

Here’s what a contract for tokenized items might look like:

</div>

<div class="listingblock">

<div class="content">

``` highlight
link:api:example$token/ERC721/GameItem.sol[role=include]
```

</div>

</div>

<div class="paragraph">

The [`ERC721URIStorage`](api:token/ERC721.html#ERC721URIStorage) contract is an implementation of ERC-721 that includes the metadata standard extensions ([`IERC721Metadata`](api:token/ERC721.html#IERC721Metadata)) as well as a mechanism for per-token metadata. That’s where the [`_setTokenURI`](api:token/ERC721.html#ERC721-_setTokenURI-uint256-string-) method comes from: we use it to store an item’s metadata.

</div>

<div class="paragraph">

Also note that, unlike ERC-20, ERC-721 lacks a `decimals` field, since each token is distinct and cannot be partitioned.

</div>

<div class="paragraph">

New items can be created:

</div>

<div class="listingblock">

<div class="content">

``` highlight
> gameItem.awardItem(playerAddress, "https://game.example/item-id-8u5h2m.json")
Transaction successful. Transaction hash: 0x...
Events emitted:
 - Transfer(0x0000000000000000000000000000000000000000, playerAddress, 7)
```

</div>

</div>

<div class="paragraph">

And the owner and metadata of each item queried:

</div>

<div class="listingblock">

<div class="content">

``` highlight
> gameItem.ownerOf(7)
playerAddress
> gameItem.tokenURI(7)
"https://game.example/item-id-8u5h2m.json"
```

</div>

</div>

<div class="paragraph">

This `tokenURI` should resolve to a JSON document that might look something like:

</div>

<div class="listingblock">

<div class="content">

``` highlight
{
    "name": "Thor's hammer",
    "description": "Mjölnir, the legendary hammer of the Norse god of thunder.",
    "image": "https://game.example/item-id-8u5h2m.png",
    "strength": 20
}
```

</div>

</div>

<div class="paragraph">

For more information about the `tokenURI` metadata JSON Schema, check out the [ERC-721 specification](https://eips.ethereum.org/EIPS/eip-721).

</div>
